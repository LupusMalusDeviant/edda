## A вҖ” Sicherheit

### A1 В· Auth-Token-Vergleich ist nicht timing-safe
**Kritisch (bei Remote-Exposure) В· S В· вң…**
`src/Edda.Hosting/Authentication/LocalAuthenticationHandler.cs:~48` vergleicht den Bearer-Token mit
`string.Equals(..., StringComparison.Ordinal)`. Das erlaubt theoretisch Timing-Seitenkanal-Angriffe.
- **Fix:** Beide Strings zu UTF-8-Bytes, Vergleich via `CryptographicOperations.FixedTimeEquals`. LГӨngen-Ungleichheit darf nicht frГјh returnen (gegen Dummy-Puffer vergleichen).
- **Akzeptanz:** Unit-Test: korrekter Token вң“, falscher Token вң—, leerer Token вң—, Token anderer LГӨnge вң—; kein `string.Equals` mehr auf dem Token-Pfad.

### A2 В· Kein Rate-Limiting auf `/api/*` und `/mcp`
**Hoch В· S В· вң…**
`src/Web/Program.cs` registriert keine Rate-Limiting-Middleware вҖ” unbegrenzte Token-Brute-Force mГ¶glich.
- **Fix:** `AddRateLimiter` (Fixed-Window, z. B. 100 Req/min/IP, konfigurierbar via `EDDA_RATE_LIMIT_PER_MINUTE`, 0 = aus) + `UseRateLimiter()`; Policy auf die `/api`- und `/mcp`-Gruppen anwenden. `.env.example` ergГӨnzen.
- **Akzeptanz:** 429 nach Limit-Гңberschreitung; Loopback-UI (Blazor-Circuit) bleibt unbeeintrГӨchtigt; Tests fГјr die Options-Bindung.

### A3 В· Neo4j startet per Default ohne Auth
**Hoch В· S В· вң…**
`docker-compose.yml` (`NEO4J_AUTH=${NEO4J_AUTH:-none}`) und `.env.example` defaulten auf No-Auth. Bei Fehlkonfiguration (Port 7687 exponiert) steht die DB offen.
- **Fix:** Install-Skripte (`install.sh`, `install.ps1`) generieren ein Zufallspasswort und schreiben es in `.env`; `.env.example`-Default auf Platzhalter mit Warnkommentar; Startup-Log-Warnung im `Neo4jGraphDatabaseProvider`, wenn `AuthTokens.None` aktiv ist.
- **Akzeptanz:** Frische Installation via Skript hat Neo4j-Passwort; bestehende `.env` mit `none` funktioniert weiter, loggt aber eine Warnung.

### A4 В· Remote-Bind ohne Auth-Token wird nicht verhindert
**Hoch В· S В· вң…**
`EDDA_BIND=0.0.0.0` ohne `EDDA_AUTH_TOKEN` exponiert API+UI unauthentifiziert; nur das Install-Skript warnt, der Code nicht.
- **Fix:** Startup-Guard in `src/Web/Program.cs`: Wenn Bind вү  Loopback **und** Token leer вҶ’ Fail-Fast mit klarer Fehlermeldung (Override-Env `EDDA_ALLOW_INSECURE_REMOTE=true` fГјr bewusste Ausnahmen).
- **Akzeptanz:** Unit-Test der Guard-Logik (reine Funktion `IsInsecureRemoteBind(bind, token, allowInsecure)`); Doku-Absatz in `docs/betrieb.md`.

### A5 В· Loopback-Erkennung hinter Reverse-Proxy umgehbar/kaputt
**Mittel В· S В· вң…**
Es gibt keine `ForwardedHeadersMiddleware`-Konfiguration. Hinter einem Proxy ist entweder jeder Request вҖһLoopback" (Proxy lokal) oder die Loopback-Erkennung falsch.
- **Fix:** `ForwardedHeaders` opt-in via `EDDA_TRUSTED_PROXIES` (kommagetrennte IPs вҶ’ `KnownProxies`); ohne Konfiguration bleiben Forwarded-Header **ignoriert** (sicherer Default).
- **Akzeptanz:** Doku in `docs/betrieb.md` (nginx/Caddy-Beispiel); Tests fГјr das Options-Parsing.

### A6 В· MCP-Token wird auch als Query-Parameter akzeptiert
**Mittel В· S В· вң…**
`src/Web/Program.cs:~58` akzeptiert `?token=` вҖ” Tokens landen in Logs, Browser-History, Referrern.
- **Fix:** Query-Parameter-Pfad entfernen; nur `Authorization: Bearer`. Falls stdio-/Legacy-Clients ihn brauchen: Deprecation-Warnung loggen statt sofort entfernen (Entscheidung im PR dokumentieren).
- **Akzeptanz:** Header-Auth вң“, Query-Auth 401 (oder Warnung); README/`docs/mcp.md` angepasst.

### A7 В· Wasm-/Subprocess-Sandbox ohne Ressourcen-Limits
**Mittel В· M В· вҡ пёҸ**
Docker-Sandbox ist gehГӨrtet (256 MB, 50 % CPU, network=none), aber der Wasm-/Subprocess-Pfad (`src/Sandboxing/Wasm/`) startet Python ohne cgroup/ulimit вҖ” DoS auf dem Host mГ¶glich.
- **Fix:** Hartes Wall-Clock-Kill (Prozess-Kill nach Timeout, existiert teils), `ProcessPriorityClass.BelowNormal`, Output-GrГ¶Гҹenlimit; unter Linux zusГӨtzlich ulimit-Wrapper. VollstГӨndige cgroup-Isolation ist out-of-scope вҖ” Restrisiko in `docs/tdk.md` dokumentieren.
- **Akzeptanz:** Endlosschleifen-Skript wird zuverlГӨssig gekillt; Test mit Fake-Runner; Doku-Absatz.

### A8 В· Edda-Container lГӨuft als root (docker.sock)
**Mittel В· M В· вҡ пёҸ**

- **Fix (pragmatisch):** Non-root-User + `group_add: [docker-gid]` in Compose; docker.sock-Mount nur im `tdk`-Compose-Profil; Betrieb ohne TDK-Docker-Sandbox lГӨuft non-root. Doku der Trade-offs.
- **Akzeptanz:** Standard-Compose lГӨuft non-root; TDK-Profil dokumentiert.

### A9 В· MCP-Token-Hashes ungesalzen (SHA-256)
**Niedrig В· S В· вң…**
`src/Security/Mcp/FileMcpTokenStore.cs:~138`. FГјr generierte High-Entropy-Tokens ausreichend, aber ohne Salt sind identische Tokens erkennbar und schwache manuelle Tokens offline-brute-forcebar.
- **Fix:** Pro Token zufГӨlliges Salt speichern, Hash = SHA-256(salt вҖ– token). Migration: Alt-EintrГӨge ohne Salt weiter akzeptieren, beim nГӨchsten Schreiben migrieren.
- **Akzeptanz:** Roundtrip-Tests neu+alt; Datei-Format-Version im JSON.

### A10 В· SecretRedactor nicht im globalen Log-/Exception-Pfad
**Mittel В· S В· вң…**
Redaction lГӨuft vor LLM-Calls, aber Exceptions (z. B. HTTP-Fehler mit API-Key in Message/URL) kГ¶nnen ungefiltert in Logs landen.
- **Fix:** Globalen Exception-Handler (bzw. die zentrale Fehler-Middleware in `Edda.Hosting`) durch `ISecretRedactor.Redact(...)` auf Message + ToString() schleusen, bevor geloggt wird.
- **Akzeptanz:** Test: Exception mit `sk-...`-Muster in Message вҶ’ Log-Ausgabe redacted.

### A11 В· HTTPS-/Proxy-Betrieb nicht dokumentiert
**Niedrig В· S В· вң…**
Kein `UseHttpsRedirection`, kein TLS вҖ” fГјr lokal ok, fГјr Remote fehlt die Anleitung.
- **Fix:** Abschnitt вҖһRemote-Betrieb hinter Reverse-Proxy (Caddy/nginx, TLS)" in `docs/betrieb.md`, inkl. Verweis auf A5-Konfiguration. Kein Code nГ¶tig.
- **Akzeptanz:** Doku-Review.

## Umsetzungsplan

Ziel: Ein schwГӨcheres Modell (z. B. Haiku-Klasse) arbeitet die Issues eigenstГӨndig ab.
Der Plan minimiert dafГјr Interpretationsspielraum: kleine HГӨppchen, jedes Issue mit
Akzeptanzkriterien, harte Arbeitsregeln, klare Eskalationspfade.

### Arbeitsregeln (gelten fГјr JEDES Issue)

1. **Ein Issue = eine Session = ein Commit.** Niemals zwei Issues mischen. Commit-Message deutsch, Format: `A1: Timing-sicherer Token-Vergleich` (Regel 10: keine KI-Mention).
2. **Vor dem Start:** betroffene Dateien vollstГӨndig lesen; prГјfen, ob der beschriebene Zustand noch stimmt (Zeilennummern sind NГӨherungen). Weicht der Code stark von der Issue-Beschreibung ab вҶ’ **abbrechen und melden**, nicht raten.
3. **CLAUDE.md-Regeln sind absolut:** Interface-First (neue Services вҶ’ Interface in `src/Core`), `IFileSystem` statt File-I/O, `TimeProvider` statt `DateTime.UtcNow`, Tools werfen nie Exceptions (`ToolResult.Fail`), userId/tenantId aus Context вҖ” nie aus Tool-Argumenten, 100 % Unit-Tests fГјr neue Klassen (Benennung `MethodName_Scenario_ExpectedResult`), XML-Doku englisch, externe Doku deutsch, keine CDN-Assets.
4. **Nach jedem Issue:** `dotnet build Edda.slnx` und `dotnet test Edda.slnx` вҖ” beides muss grГјn sein (0 Warnings, TreatWarningsAsErrors). Rot = selbst fixen oder Commit verwerfen; niemals Tests lГ¶schen/skippen, um grГјn zu werden.

6. **Nicht anfassen:** MCP-Default-Deny-Logik lockern, Security-Layer entfernen/umgehen, Namespaces umbenennen, вҖһnebenbei" refactoren.
7. **Eskalieren statt improvisieren:** Wenn ein Akzeptanzkriterium nicht erreichbar scheint, das Issue mit BegrГјndung zurГјckgeben. Teilfortschritt committen ist ok, wenn Build+Tests grГјn sind und der Rest als TODO im Issue (nicht im Code) dokumentiert wird.
